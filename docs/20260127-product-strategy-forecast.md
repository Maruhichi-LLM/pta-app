# Knot サービス プロダクト思想・モジュール評価・売上/原価予想

**作成日**: 2026-01-27  
**対象**: /web (Next.js 16.1.3 / Prisma 5.21.1)  
**参照**: `docs/20260126-evaluation.md`

> 注記: 本書の売上・原価は実数ではなく、現時点の仮定に基づく試算です。数値は可変パラメータとして扱い、事業計画で再調整してください。

---

## 1. プロダクト思想 (Product Philosophy)

1. **意思決定の透明化**
   - チャットを起点に全ての成果物へ紐付け、意思決定の経緯を追跡可能にする。

2. **会計の信頼性と実務性**
   - 多段階承認、年度締め、自動集計を標準搭載し、団体運営の信頼性を担保する。

3. **団体の規模と成熟度に合わせたモジュラー設計**
   - 必要機能だけを有効化し、最小構成から段階的に拡張できる。

4. **日本の非営利運営に最適化**
   - 会計年度、役割慣行、日本語UIを前提に現場負荷を下げる。

5. **トレーサビリティと監査性の内蔵**
   - 監査ログ、内部統制チェック、履歴保存によりガバナンスを標準化。

## 2. モジュール評価と技術的まとめ (全モジュール)

> 評価は 5.0 満点の主観スコアです。MVPとしての完成度・信頼性・拡張性を総合評価しています。

### 2.1 コアモジュール

#### Knot Chat (`/chat`, `/threads/[threadId]`) — **4.2 / 5**
- 役割: 意思決定のハブ。スレッド起点でToDo/会計/文書へ変換。
- 技術: `ChatThread`, `ChatMessage` + `sourceThreadId/sourceChatMessageId` により追跡性を確保。
- 留意: リアルタイム未対応。通知が未整備。

#### Knot ToDo (`/todo`) — **4.0 / 5**
- 役割: 会話起点のタスク化と進捗管理。
- 技術: `TodoItem` とチャットの連携、担当/期限/状態管理。
- 留意: 一括更新・リマインドなどの運用機能は未整備。

#### Knot Event (`/events`, `/events/[id]`) — **4.2 / 5**
- 役割: イベント作成、出欠管理、CSV/PDF出力。
- 技術: 出欠集計とPDF生成 (pdfkit) を備える。
- 留意: 権限粒度や通知が限定的。

#### Knot Calendar (`/calendar`) — **3.8 / 5**
- 役割: 団体イベント + 個人予定の月次統合ビュー。
- 技術: `PersonalEvent` と団体イベントの統合表示、SSR。
- 留意: 週表示や外部カレンダー連携は未実装。

#### Knot Accounting (`/accounting`) — **4.6 / 5**
- 役割: 団体会計の中核。承認・予算・年度締めまで一体化。
- 技術: `Ledger`, `Approval`, `Account`, `Budget`, `FiscalYearClose` など会計ドメインを網羅。
- 留意: 監査・テスト不足はリスクだが機能完成度は高い。

#### Knot Document (`/documents`, `/documents/[docId]`) — **4.0 / 5**
- 役割: 文書保管・バージョン管理・カテゴリ分類。
- 技術: `Document`, `DocumentVersion`, メタデータ + ストレージ連携。
- 留意: 検索性や権限の粒度は拡張余地あり。

#### Knot Management (`/management`) — **4.0 / 5**
- 役割: メンバー招待、役割管理、団体設定。
- 技術: 招待コード + RBAC (管理者/会計係/監査役/メンバー)。
- 留意: 役割粒度は限定的。リソース単位の権限制御が今後の課題。

#### Knot Store (`/store`) — **3.9 / 5**
- 役割: モジュールの有効/無効を管理。
- 技術: `enabledModules` の管理、system module 常時有効化。
- 留意: 課金連動やプラン管理は未実装。

#### Knot Export (`/export`) — **3.7 / 5**
- 役割: イベント出欠や収支計算書のCSV/PDFエクスポート。
- 技術: `event-exports`, `accounting-exports` API を利用。
- 留意: 出力形式の多様化・自動定期出力は未対応。

### 2.2 拡張/ガバナンス系モジュール

#### Knot Workflow (`/workflow`) — **3.8 / 5**
- 役割: 汎用の申請・承認フローを構成。
- 技術: `ApprovalRoute`, `ApprovalStep`, `ApprovalTemplate`, `ApprovalApplication`, `ApprovalAssignment`。
- 特徴: JSONスキーマ型フォーム定義、段階承認、ロールベース承認。
- 留意: 通知/期限管理が未整備。UIの導線強化が必要。

#### Knot Audit (`/audit`) — **3.9 / 5**
- 役割: 監査ログ、内部統制チェック、指摘事項管理。
- 技術: `AuditLog`, `AuditFinding`, `InternalControlRule` + 自動ルール検知。
- 特徴: 職務分掌/高額承認/出所リンク欠落などのチェック。
- 留意: 監査レポート出力や外部監査向け機能は未整備。

#### Event Budget Extension (`/events/.../budget`) — **4.1 / 5**
- 役割: イベント単位の予算/実績管理と会計への取り込み。
- 技術: `EventBudget`, `EventTransaction`, `EventBudgetImport`。
- 留意: 予算更新の履歴管理は今後の強化ポイント。

## 3. 売上予想 (Revenue Forecast)

- 価格プラン (仮置き)

  - Standard: ¥3,000 / 月 / 団体
  - Premium: ¥8,000 / 月 / 団体
  - Enterprise: ¥50,000 / 月 / 団体 (オンプレ/専任対応想定)
- 対象市場: 国内非営利団体 (約 430,000 団体)
- 収益はサブスクのMRRを基準に算出

## 4. 原価予想 (COGS / サービス原価)

### 4.1 原価の定義 (SaaS想定)
- インフラ: アプリ/DB/オブジェクトストレージ/バックアップ
- 通信/配信: メール送信、PDF/CSVエクスポート
- 決済手数料: 月次課金の手数料
- サポート運用: 有料団体への一次対応

### 4.2 仮定値 (例)
- 固定費 (基盤): **¥250,000 / 月**
- 変動費 (1団体あたり): **¥1,300 / 月**
  - インフラ変動: ¥500
  - サポート: ¥800
- 決済手数料: **売上の 3.6%**

### 4.3 12か月後の原価試算 (MRRベース)

| シナリオ | 有料団体数 | MRR | 原価 (概算) | 粗利 | 粗利率 |
|---|---:|---:|---:|---:|---:|
| Conservative | 185 | ¥940,000 | ¥524,340 | ¥415,660 | 約44% |
| Base | 490 | ¥2,340,000 | ¥971,240 | ¥1,368,760 | 約58% |
| Aggressive | 1,230 | ¥6,100,000 | ¥2,068,600 | ¥4,031,400 | 約66% |

> 補足: 固定費・変動費は実運用のログ (同時接続数/ストレージ消費/ジョブ実行回数) で再調整が必要。

## 5. 相談用ポイント (ChatGPTで検討すべき論点)

1. **価格戦略の再設計**
   - 団体規模別に段階的課金 (人数・ストレージ・承認フロー数など)。

2. **原価構造の最適化**
   - ストレージとPDF生成のバッチ化・アーカイブ化で変動費を抑える。

3. **導入拡大の優先セグメント**
   - NPO法人 vs 町内会 vs PTA のどこから攻めるか。

4. **運用負荷の見積もり**
   - サポート体制やFAQ整備が粗利に与える影響。

5. **監査モジュールの価値訴求**
   - 監査・内部統制を「差別化ポイント」として売るか。
